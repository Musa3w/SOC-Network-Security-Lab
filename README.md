# 🛡️ Enterprise SOC & Network Security Laboratory

This repository showcases an Enterprise-grade Security Operations Center (SOC) and Network Security lab built entirely from scratch in a local virtualized environment (VMware Workstation). The architecture is strictly designed as an air-gapped, isolated network to simulate and analyze corporate cyber threats safely.

## 🏗️ 1. Architecture & Network Segmentation

* **Environment:** Built locally inside an isolated virtual switch network via VMware Workstation.
* **Network Topology:** * **OPNsense Firewall:** Acts as the core gateway. The `WAN` interface faces a simulated external network, while the `LAN` interface secures the internal corporate network.
  * **Active Directory (Windows Server DC)** & **Windows 10 Client:** Resides within the same internal subnet, handling centralized domain services and enterprise endpoint management.

### 📋 Infrastructure Evidence
* **`windows_server_dc.png`** — Demonstrates the domain backbone with active roles: Active Directory Domain Services (AD DS), DHCP, and DNS, managing the `APPLE.LOCAL` forest.
* **`windows_10_domain.png`** — Confirms the client workstation (`CLIENT-10`) is successfully joined to the `APPLE.LOCAL` domain.

## ⚙️ 2. Log Management & Integration

All security events and raw telemetry generated by network gateways and corporate endpoints are centrally ingested into a SIEM platform.

### 🔍 Log Parsing & Indexing
Raw, comma-separated Syslog payloads transmitted by OPNsense are systematically normalized inside Splunk using custom field extraction rules.

### 📋 Log Processing Evidence
* **`splunk_field_extraction.png`** — Displays the Splunk Interactive Field Extractor (IFX) capturing raw log strings to map critical security indicators.
* **`splunk_parsed_fields.png`** — Proves successful parsing. Firewall events such as `block` actions, ingress interfaces (`em1`), and TCP SYN flags (`tcp_flag = S`) are cleanly indexed into structured columns.

## 🚨 3. Threat Detection & Defense

Threat simulations are conducted using a dedicated **Kali Linux** attacker machine.

* **Simulation:** Unauthorized connection requests, network sweeps, and port scans are launched from the Kali node (`192.168.1.50`) targeting protected internal resources.
* **Defense Mechanism:** Stringent segmentation and explicit drop rules implemented on OPNsense instantly restrict malicious or unauthenticated traffic.

### 📋 Attack & Defense Evidence
* **`kali_attack.png`** — Shows the attacker's IP configuration via `ip a` and the resulting connection timeout when attempting to probe restricted interfaces.
* **`opnsense_live_blocks.png`** — OPNsense Firewall Live View validating that TCP requests originating from `192.168.1.50` are properly dropped (`block`).
* **`opnsense_firewall_rules.png`** — Confirms the strict enforcement of the `Block Others from GUI` rule inside the OPNsense LAN firewall rules interface.

## 🛠️ Deployment Instructions / How to Replicate

1. Set up an isolated virtual switch/network segment in your hypervisor (VMware/VirtualBox).
2. Install and configure **OPNsense** with dual interfaces (WAN/LAN).
3. Deploy **Windows Server**, promote it to a Domain Controller (AD DS, DNS, DHCP), and configure a workstation client joined to the domain.
4. Deploy **Splunk Enterprise** (SIEM) and ingest Syslog data coming from your OPNsense gateway.
5. Create custom Regex Field Extractions for normalized data structuring.
6. Launch test traffic from **Kali Linux** and verify blocking behaviors on the firewall Live View and SIEM dashboards.
